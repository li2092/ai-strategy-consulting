# Checkpoint 协议 · 边写边审

> 这份文档解决一个核心矛盾:**默认"一次跑完"模式让一次性出令人满意的报告几乎不可能**。
>
> Checkpoint 协议把"写完审"改成"写时即审"——AI 每完成一节/一章,主动暴露问题让用户审,用户说继续才进下一节。

---

## 0 · 为什么 Checkpoint 是强制的

### 0.1 一次跑完模式的失败模式

让 AI 一气跑完一篇 30 页报告,几乎必然出现以下问题之一:
- **质量曲线衰减**:第 1 章用心写 + 引用 + 论证,第 5 章变成 bullet 堆砌
- **战略漂移**:写到第 4 章发现和第 1 章的判断不一致,但已经写完不想改
- **隐藏失败**:某条硬约束 ✗,AI 不主动报告,等用户发现已经太晚
- **方向错误**:用户其实想要 A 方向,AI 跑完才发现写的是 B 方向

### 0.2 Checkpoint 模式的设计

每完成一个**最小可审单元**(Step 1 一节 / Step 2 一章),AI **主动停下来**做三件事:

1. 跑硬约束自查(H1-H5 每条 ✓/✗)
2. 报告自己的疑虑(没疑虑写"无")
3. 列需要用户确认的问题(1-2 个)

用户回应 `继续` 之前,AI **不允许写下一节/章**。

这是流程级强制,不是建议。

---

## 1 · Checkpoint 触发时点

| 时点 | 触发条件 |
|------|---------|
| Step 1.1 完成 | "行业速览(含前沿对标)"写完 |
| Step 1.2 完成 | "全量 AI 应用候选 12-20 个"写完 |
| Step 1.3 完成 | "Top 3 推荐 + 不做什么"写完 |
| Step 2 每章完成 | 每写完一章 |
| 硬约束连续 2 次 ✗ | AI 主动触发,不等节完成 |
| AI 遇到不知道的关键事实 | AI 主动触发,问用户而不是猜 |

**绝对禁止跨过 Checkpoint 写下一节**。

## 2 · Checkpoint 固定格式

下面是必须严格遵守的模板。**不能省、不能改、不能精简**。

```markdown
---
## ⏸ Checkpoint · [节名/章名]

### 硬约束自查

| # | 约束 | 实际 | 目标 | ✓/✗ |
|---|------|------|------|-----|
| H1 | 表格字数占比 | __% | ≤ 25% | ✓/✗ |
| H2 | 散文段数(每段 150-300 字) | __ 段 | ≥ 2 | ✓/✗ |
| H3 | 具象化锚点(仅 Top 推荐适用) | __ 个 | ≥ 1/方向 | ✓/✗/N/A |
| H4 | 外部对标(仅 Top 推荐适用) | __ 个 | ≥ 1/方向 | ✓/✗/N/A |
| H5 | 取舍段(仅 Top 推荐适用) | __ 个 | ≥ 1/方向 | ✓/✗/N/A |

### 我的疑虑(主动暴露 1-3 条,无则写"无")

- [疑虑 1,要具体到"哪段哪个判断"]
- [疑虑 2]
- [疑虑 3]

### 需你确认(1-2 个具体问题)

1. [是否进入下一节? / 当前内容是否需要调整方向?]
2. [可选:某个具体争议点的判断]

---
```

### 2.1 硬约束自查的诚实纪律

- **数字必须真算**,不能拍脑袋。表格占比要数字数 / 总字数,不能写"约 20%"
- **✓/✗ 必须真打**,不允许把 ✗ 写成 ✓
- **N/A 仅适用**:H3-H5 在不涉及推荐方向的章节(如"背景与问题诊断")可标 N/A;但只要章节里出现了推荐,就必须满足

### 2.2 疑虑的写法

**好疑虑**(具体、可决策):
- "Top 1 的对标我引用了 LinkedIn 2024 年的数据,但只找到二手报道,数字 1.5 小时/天 是否需要你确认或换对标?"
- "候选方向 X 和候选方向 Y 的技术路径其实重合,要不要合并成一个?"
- "本节字数压到 800 字,但 H4 对标我只找到 1 个,是否要扩到 2 个?"

**差疑虑**(空泛、不可决策):
- "可能还有其他角度需要考虑"(没说什么角度)
- "需要进一步研究"(没说研究什么)
- "等待用户反馈"(没问具体问题)

差的疑虑 = 没有疑虑 = 显得 AI 在敷衍。

### 2.3 需你确认的问法

每次 1-2 个**具体的、可一句话回答的**问题。例:
- "是否进入 Step 1.3?"(回答 yes/no/调整)
- "Top 3 是否需要把候选 X 加进来?"(回答 加/不加)
- "本章对标 LinkedIn 是否合适,还是换 Google?"(回答 LinkedIn/Google)

避免:
- "是否还需要补充?"(太泛)
- "你怎么看?"(没问点)
- "需要我再写一遍吗?"(应该自己判断要不要写)

## 3 · 用户响应协议

| 用户输入 | AI 行为 |
|---------|--------|
| `继续` / `ok` / `进入下一节` / `下一章` / `好` | AI 进入下一节/章,不重写当前 |
| `改 [具体点]` / `把 X 改成 Y` | AI 只改指定部分,不触发全文重写;改完重新 Checkpoint |
| `重写本章` / `推倒重来` | AI 重写整章,改完重新 Checkpoint |
| `停` / `暂停` / `等等` | AI 挂起,不再推进,等下次指令 |
| 用户给新内容 / 新数据 / 新方向 | AI 吸收,更新当前节,重新 Checkpoint |
| 用户没回应(超过两轮对话仍没说继续) | AI 应主动问"是否继续?",不要自动推进 |

### 3.1 绝对禁止

- 跳过 Checkpoint 直接写下一节(包括"为了节省你的时间,我直接...")
- 把硬约束 ✗ 写成 ✓ 或者隐藏
- 用户没明确说"继续",AI 自己解读"应该是同意了"
- 把疑虑藏起来不报告(以为说出来显得不专业)

### 3.2 用户偷懒的反纪律

如果用户在 Checkpoint 时说"你看着办继续吧" / "都行,继续":

```
我建议在以下几点你做出决定再继续,不然写到后面发现方向偏了重写成本高:
1. [关键决策点 1]
2. [关键决策点 2]

你 30 秒决定一下,我等。
```

**不要默认用户"看着办"= 同意继续,要把决策权抛回去**。战略报告的方向判断必须用户做,AI 只能给候选。

## 4 · 异常处理

### 4.1 AI 不知道关键事实

AI 写到一半发现需要的事实不知道(如"客户当前广告业务收入"),处理:

```
---
## ⏸ Checkpoint · [节名] · 触发原因:关键事实缺失

### 缺什么
[具体说明缺的事实,例:"腾讯广告业务 2024 年营收占公司总营收的比例"]

### 为什么需要
[说明这个事实如何影响后续判断]

### 我建议的处理
- 选项 A:你提供这个数字 → 我继续写
- 选项 B:你确认我可以用【推断】标签估算 → 我用合理范围(如 "10-15%【推断】")继续
- 选项 C:这个数字其实不影响推荐方向,可以不写 → 我跳过

请选 A/B/C。
---
```

**绝对禁止**:AI 自己编一个数字塞进去。

### 4.2 硬约束反复达不到

某条硬约束 AI 已经尝试 2 次仍 ✗:

```
---
## ⏸ Checkpoint · 硬约束反复达不到

### 反复 ✗ 的约束
H4 外部对标(已尝试 2 次)

### 我尝试过的
1. 第 1 次:对标 LinkedIn,但只找到二手报道
2. 第 2 次:对标 Google Ads,但找不到具体产品名

### 我的判断
该方向可能不成熟,世界前沿没有公开案例 → 建议从 Top 3 中删除

### 需你确认
是否同意删除该方向?或者你有其他对标线索?
---
```

**绝对禁止**:AI 凑合写一个不合格的对标过关。

### 4.3 用户的判断和 AI 的硬约束冲突

用户说"我不想要这么多对标,简单点":

```
理解。但 H4 外部对标是 skill 的核心硬约束,直接简化会失去战略报告
对标世界前沿的可信度。

我建议:
- 折衷 A:对标精简到每方向 1 个但要齐 4 要素(公司+产品+时间+效果)
- 折衷 B:你接受去掉 H4(承担可信度下降)

请选,或给我新指示。
```

**绝对禁止**:AI 自动绕过硬约束。约束的修改必须用户明确同意。

## 5 · Checkpoint 真实范例

### 5.1 好范例

```markdown
---
## ⏸ Checkpoint · Step 1.2 全量 AI 应用候选

### 硬约束自查

| # | 约束 | 实际 | 目标 | ✓/✗ |
|---|------|------|------|-----|
| H1 | 表格字数占比 | 31% | ≤ 25% | ✗ |
| H2 | 散文段数 | 3 段 | ≥ 2 | ✓ |
| H3 | 具象化锚点 | N/A(本节列候选,Top 3 在 1.3 章) | ≥ 1/方向 | N/A |
| H4 | 外部对标 | N/A | ≥ 1/方向 | N/A |
| H5 | 取舍段 | N/A | ≥ 1/方向 | N/A |

H1 超标:本节列了 18 个候选 + 五维评分表,表格行数较多。我已经
重写过一次,把候选评分用紧凑表 + 论述拆开,占比从 45% 降到 31%。
要不要我再裁:把"五维评分"放到附录,主表只留候选名+一句话?

### 我的疑虑

1. 候选 #11(广告主自助 Agent)和候选 #14(代理商工具栈 AI 化)
   其实是一个方向的不同切片,要不要合并?
2. 我没列"内部研发流程 AI 化"(代码 Copilot 之类)候选,因为这
   不是广告业务专属。要不要补?
3. 候选 #5、#7、#9 都涉及"创意素材生成",但腾讯广告核心客户对
   素材调性把控严,这一类我有点犹豫要不要全砍掉。

### 需你确认

1. H1 超标我是否继续重写裁到 25% 以内?(yes 继续裁 / no 接受)
2. 是否把 #11 和 #14 合并?
---
```

### 5.2 差范例(应该被立即驳回)

```markdown
## Checkpoint
H1-H5 都通过。
有一些细节可以进一步完善。
请确认是否继续。
```

差在:
- 没数字(✓ 是怎么来的)
- 疑虑空泛("有一些细节"没说什么)
- 问题不具体("是否继续"没说继续什么)

## 6 · Checkpoint 的频率与节奏

### 6.1 推荐节奏

| 阶段 | Checkpoint 数 | 时间预算 |
|------|--------------|---------|
| Step 1(全程) | 3 次(1.1/1.2/1.3 各一次) | AI 30-60 分钟生成 + 用户 30 分钟审 = 0.5 天 |
| Step 2(15-40 页报告) | 7-9 次(每章一次) | AI 1-2 小时生成 + 用户 1-2 天审 = 1-2 天 |

总 Checkpoint 数:10-12 次。每次用户花 5-15 分钟审 = 总共 1-3 小时审稿时间。

### 6.2 异常密集 Checkpoint

如果出现以下情形,Checkpoint 频率会上升:
- 硬约束反复 ✗
- 关键事实缺失
- 用户给新指示需要重定方向

这是正常的——边写边对齐比写完再返工成本低得多。

---

## 7 · 用户也要遵守的纪律

Checkpoint 协议是双向的。**用户的责任**:

1. **不偷懒**:别说"你看着办",要做决策
2. **不模糊**:回应要具体到"yes/no" 或 "改 X 成 Y",不说"我觉得可能..."
3. **及时**:Checkpoint 提出的问题最好 30 分钟内回(否则上下文会冷)
4. **诚实承担风险**:如果用户说"接受 H4 ✗",AI 跟进时会标"H4 ✗(用户已认可)",事后报告质量问题不能怪 AI

这些纪律是为了让"AI 边写边审"真正发挥价值——如果用户也偷懒,Checkpoint 就退化成形式。

---

## 8 · 实施总结

> Checkpoint 协议把"AI 主导生产 + 用户事后审" 改成 "AI 持续暴露 + 用户持续决策"。
>
> 这个协议的核心是把质量控制权下放给"持续小决策",而不是集中在"最终大审稿"。

不实施 Checkpoint → 一次跑完 → 大量返工 → 用户怀疑 skill。

实施 Checkpoint → 边写边审 → 单点小返工 → 用户对齐感强。

skill 的所有其他设计(2 步主流程 / 5 项硬约束 / 反模式速查)都依赖 Checkpoint 协议生效。**没有 Checkpoint,其他设计都形同虚设**。

# AI 战略报告反模式速查

> 沉淀自多轮外审 + 实战踩坑。每个反模式列触发条件、典型表现、识别方法、修订方案。
>
> **写之前看**:避免新写出反模式;**写完后看**:用 grep + 人工二次确认。
> 本文件本身大量举"反模式表述"作为反例(例如"千份级文件验证"),grep 时**会被预期命中**——客户实际交付稿件应保持 0 命中。

## 目录

| 类别 | 含义 | 数量 | 处理 |
|---|---|---|---|
| P0 类 · 硬错 | 事实错 / 保密泄漏 / 法律风险 | 3 项 | 必删 + 改写 |
| P1 类 · 论证越级 | 推理链断裂 / 单源越级 / 口径不闭合 | 3 项 | 必改 |
| P2 类 · 战略深度不足 | 全表格 / 对标含糊 / 工具堆砌 / 缺取舍 / 缺疑虑 | 5 项 | 必改 |
| P3 类 · 叙事腔调失真 | 威胁恐吓 / 第二人称 / 自媒体腔 | 3 项 | 视情况修订 |
| P4 类 · 修辞与篇幅 | AI 腔 / 排比设问 / 数量不一致 | 3 项 | 视情况修订 |

## P0 类 · 硬错

### P0-1 · 编造组织关系 / 资质 / 引文

**触发条件**:写客户背景、对手分析、国际公司引文时缺少官方文件支撑。

**典型表现**:
- 把"X 集团 → Y 子公司 → Z 客户"的祖孙股权链写成"X 集团是 Z 客户的天然牵头单位"
- 编造"客户参与了 X 国标制定"等未经验证的资历
- "某国际公司官方说: '<虚构的英文原文>'"——无法在该公司官方文档核到的引文
- "X 公司提升效率 47.3%"——无来源的精确百分比

**识别方法**:
```bash
# 高风险字眼
grep -nE "母集团|祖集团|牵头单位|天然承担者|参与了.*国标|主导.*标准" 主报告.md
# 精确百分比 + 检查是否有来源
grep -nE "[0-9]+\.[0-9]+%" 主报告.md
# 英文引文
grep -nE "原文.*['\"].*['\"]" 主报告.md
```

**修订方案**:
- 股权 / 资质 / 引文 / 数字必须有官方文件或学术论文支撑——查不到就**删除**,不要"听说"叙事
- 精确百分比改为区间:"47.3%" → "30-50%【推断】"
- 客户名匿名化:"某航司" → "国内某大型枢纽机场"
- 引文必须能在官方文档核到,否则改为概括:"X 公司官方文档将本体定位为... 的语义基础层"

**实战案例**:
- 曾把"A 子公司 + B 通用模型"的局部合作外推成"C 集团 → D 母公司 → E 客户"祖孙股权链(实际控股链路在另一条),被外审判 P0
- 同类编造"客户参与 X 国标"——CLAUDE.md 铁律 6 的重犯

### P0-2 · 跨客户内容入稿

**触发条件**:项目组同时服务多个客户,有跨客户的工具 / 数据 / 经验。

**典型表现**:
- 在客户 B 报告中写"客户 A 已验证"、"客户 A 已开发的 X skill"
- "千份级文件实战验证"、"项目组在 X 行业已积累 N 个案例"——隐含跨客户身份
- 把项目组在 A 客户的 KPI 当 B 客户战略的"已验证"基础
- 项目组自己的公开活动(演讲 / 博客 / 文章)当"行业共识"的论据

**识别方法**:
```bash
grep -nE "(已验证|实战验证|已开发的|内部已用)" 主报告.md
grep -nE "(我们|本项目组|已服务|已积累).*[0-9]+ ?个" 主报告.md
```

**修订方案**:
- 删去具名客户与工具,**不要**用"项目组已积累的 X 经验"做替代(这仍是自证)
- 改用三类独立来源:① 公开学术论文 / ② 国际或国内标杆公司公开案例 / ③ 政府部委文件
- 工具栈改为开源生态描述("开源本体工程方案 / 主流 RAG 框架"),不绑具名工具
- 数量描述改虚:"多家"、"若干家"

**写之前的三问自检**:
1. 这段证据是否属于本客户?
2. 是否独立于项目组的外部来源?
3. 如果交付给第三家客户,这段话还站得住吗?

任一"否" → 删除或改写。

**实战案例**:曾在某客户主稿中把"客户 A 已验证"、"客户 A 的某专项 skill"、"千份级文件实战验证"当客户 B 论据;同稿还把"项目组成员在某行业研学会做主题分享"当方法论可行性依据——被外审连判两个 P0,均为同类二次重犯。

### P0-3 · 单一事实源未闭环(活口径残留)

**触发条件**:报告主稿改了,但底稿(ROI 测算 / 附件 / 入口章节)里旧口径还在。

**典型表现**:
- 同一文件同时说"已作废"又说"正式对外口径以 X 章为准"
- 主报告 §6 写新数字,附件 A1 写旧数字
- 主稿改了,但封面 / 目录 / 第 0 章还引用旧文件号
- 多文档相互引用,引用链断裂

**识别方法**:
```bash
grep -nE "(已作废|历史版本|不作为对外).*?(正式对外|主口径|当前口径)" 全部文档/*.md
grep -nE "(投入|回报|ROI).*[0-9]+ ?万" 全部文档/*.md
```

**修订方案**:
- 主稿改后**回扫所有底稿**(咨询审稿 6 步法 · 第 6 步)
- 作废章节加【已作废】标签 + 明示新口径所在文件与章节
- 文件号、章节号、数字三类一致性必须二次 grep 确认

## P1 类 · 论证越级

### P1-1 · 引文越级(单源 → 行业必做)

**触发条件**:引用某家国际公司的方法论时上升为"行业必做"。

**典型表现**:
- "某国际厂商在过去十年中反复论证: 本体是企业 AI 的核心瓶颈"——然后推出"X 客户首期必须建本体层"
- "某 AI 公司 CEO 说 Agent 是未来——所以 X 客户必须现在做 Agent"

**修订方案**:
- 引用降级为"借鉴"而非"必做"
- 客户首期方案降级为"增强实验"或"条件启动",附触发条件
- 增加"方法论应用边界"段落,明确"借鉴 ≠ 复制"

**实战案例**:某稿因单家国际厂商方法论引文 + 子模块"必做"结论被判 P1。修订后改为"条件启动",触发条件"Week 5-6 纯 RAG 查准率 < N%"——把不可证伪的方法论判断转为可观测的工程门槛。

### P1-2 · 战略判断粗化(路径被堵死类表达)

**触发条件**:写竞争分析或技术路线判断时偷懒。

**典型表现**:
- "AI 大模型路径被大厂堵死了"
- "X 行业已经被 Y 覆盖完了"
- "市场没机会了"

**识别方法**:
```bash
grep -nE "(被堵死|被堵|没机会|覆盖完|路径被堵)" 主报告.md
```

**修订方案**:
- 必须分至少三层(基座 / 行业 / 应用)
- 改为:"基座层已由 X 占位,行业层 Y 子赛道仍有空间,应用层 Z 类型客户尚未触达"
- 同时给"什么条件下空间会消失"——可证伪的判断才有战略价值

### P1-3 · 自证式外部论据(项目组活动当行业共识)

**触发条件**:项目组成员有公开活动(演讲、博客、文章、开源 repo)。

**典型表现**:
- 把"本项目组顾问在 X 大会的分享"作为"方法论可行性"的外部论据
- 用项目组博客文章佐证"行业共识"

**修订方案**:
- 删除自证式来源
- 改用(按可信度排序):① 政府官方文件 / ② 国际公司官方文档 / ③ 学术顶会论文 / ④ 第三方研究报告(BCG / McKinsey / IDC / Gartner)/ ⑤ 公开客户案例

## P2 类 · 战略深度不足(本次重点强化)

> M3-M4-M6 矛盾对应的反模式集群。AI 容易给"看似完整、实则空洞"的战略——这一类反模式比 AI 腔更伤稿件可信度。

### P2-1 · 全表格章节(缺散文论证)

**触发条件**:章节内容超过 60% 都是表格 / 列表 / bullet。

**典型表现**:
- 整章只有"对标表 + 推荐表 + 路线图表"三张表,没有判断和论证
- 表格之间没有过渡段
- 表格罗列了 5 个对手,但没有任何一段散文说"所以我们差在哪 / 我们抢什么"

**识别方法**:目测 + 字数估算。一章里散文段(连续完整段落,非 bullet)占比 < 25% 就触发。

**修订方案**:
- 强制硬约束 H1: 表格 ≤ 25%
- 强制硬约束 H2: 每章至少 2 段连续散文(每段 100-200 字),给"判断 + 论证"
- 表格只承担"罗列 / 对照 / 总览";结论和取舍必须用散文写
- 详见 [03-strategy-depth.md §H1-H2](03-strategy-depth.md)

**反例**:
> ## 4 · Top 推荐
> | 方向 | 价值 | 难度 |
> |---|---|---|
> | A | 高 | 中 |
> | B | 高 | 高 |
> | C | 中 | 低 |
>
> ## 5 · 路线图
> | Phase | 时间 | 交付 |
> |---|---|---|
> | 1 | M1-M3 | MVP |
> ...

通篇没有一段散文解释"为什么 A 排第一"、"B 高难度具体难在哪"——这就是全表格章节。

### P2-2 · 对标含糊(只有公司名,没有"产品+时间+规模+来源")

**触发条件**:写国际或行业对标段落时。

**典型表现**:
- "Google 用 AI 改造了搜索"(没产品名 / 没时间 / 没规模)
- "Palantir 在政府客户做得很好"(没具体客户 / 没数字)
- "OpenAI 的 Operator 很厉害"(没用例 / 没限制条件)

**修订方案** —— 强制硬约束 H4 对标格式:
```
[公司]+[产品/项目]+[时间]+[规模或可观测指标]+[来源]
```

**正例**:
> Google 在 2024 Q4 上线 NotebookLM Audio Overview,自动把上传材料合成两人播客对话(2-15 分钟),Google Labs 公开数据用户使用次数超 X 万——为"长文档→音频"开辟了新落地形态。【来源: Google Labs Blog 2024-09 + TechCrunch 2024-10】

**反例对照**:
> Google 在音频 AI 上很有突破。【含糊,无产品 / 无时间 / 无规模】

详见 [03-strategy-depth.md §H4](03-strategy-depth.md)。

### P2-3 · 工具堆砌(堆 RAG / Agent / MCP 名词,不写能替代什么任务)

**触发条件**:写 AI 战略主线、技术路线、Top 推荐时。

**典型表现**:
- "采用 RAG + Agent + MCP + Function Calling 构建智能 Copilot"——读完不知道这能做什么
- "基于多模态大模型 + 知识图谱 + 工具调用"——没有具体的输入 / 输出 / 替代了什么人工
- 用 5 个技术名词描述一个本来用 2 句话能说清的功能

**修订方案** —— 强制硬约束 H3 具象化锚点格式:
```
具体岗位/场景:
- 输入: [具体什么数据/文件/请求]
- AI 处理: [具体哪个模型/什么 prompt/调用什么工具]
- 输出: [具体什么形态——报告/邮件/数据/代码]
- 替代了: [原来需要 X 小时的 Y 任务]
- 保留人做的: [质量审核/最终决策/客户沟通]
```

**正例**:
> **法务合同审阅 Copilot**
> - 输入: 上传 PDF 合同(平均 30 页)
> - AI 处理: GPT-4 + Anthropic Contracts library 抽取关键条款 + 与公司模板做差异比对
> - 输出: 风险点清单 + 修订建议 markdown
> - 替代了: 法务初审(原 4-6 小时)→ AI 30 秒
> - 保留人做的: 高风险条款的最终判断 + 与对方法务的谈判

**反例**:
> 基于 RAG + Agent 构建合同 AI Copilot。【堆名词,不知道能做什么】

详见 [03-strategy-depth.md §H3](03-strategy-depth.md)。

### P2-4 · 只有推荐,没有取舍(缺"不做什么")

**触发条件**:写 Top 推荐 / 路线图章节。

**典型表现**:
- 列了 3-5 个推荐方向,每个都说"价值高、难度可控"
- 通篇没有一段说"以下方向我们刻意不做,原因是 X"
- 给客户的感觉是"什么都好,什么都做"——这就是没取舍

**修订方案** —— 强制硬约束 H5 取舍段(三选一):
```
方案 A: 直接给"不做什么"清单
- 候选方向 X: 不做。原因: [具体取舍逻辑]
- 候选方向 Y: 不做。原因: [具体取舍逻辑]

方案 B: 给"做 vs 不做"对照表 + 散文解释
| 方向 | 做/不做 | 原因 |

方案 C: 在每个推荐里写"代价"
推荐方向 X: ...
代价: 不能同时做 Y; 需牺牲 Z 资源
```

**好范例**:
> ## 5 · 不做什么
> 在 12 个候选方向里,以下 3 个是经过权衡后的明确取舍:
>
> **不做: 自研基础大模型**
> 国内开源模型(Qwen / DeepSeek / 智谱)已能满足客户场景,自研 70B+ 模型需 2000 万 GPU 投入和 10+ 算法团队——超出客户付费能力。该方向交给基座层公司,我们集中在应用层。
>
> **不做: 全员 AI 培训**
> 培训需求看似普世,但短期内 AI 工具迭代速度过快(每季度更换),培训内容半年就过时。改为:为 Top 5 战略方向的核心团队配置 1 对 1 教练。
>
> **不做: 通用 AI 客服**
> 客户行业是 toB 工程基建,客服量低且单次价值高,通用客服效果差。该方向 ROI 不成立。

详见 [03-strategy-depth.md §H5](03-strategy-depth.md)。

### P2-5 · 只有方案,没有疑虑(AI 全程自说自话)

**触发条件**:每个章节写完都很"自信",AI 不主动暴露任何风险或不确定。

**典型表现**:
- 通篇没有一段"我不确定 X / 这里我做了 Y 假设 / 如果你的客户是 Z 类型,这个推荐就要换"
- 每个推荐都打满分
- 风险章节只有"市场风险 / 技术风险 / 团队风险"三个万金油标题

**修订方案**:
- 强制 Checkpoint 协议: 每章写完必须主动暴露 ≥ 1 项疑虑或不确定
- 风险章节必须分客观风险(对手领先、技术不成熟)和**主观判断风险**(我假设了 X,如果错了路线全变)
- 详见 [06-checkpoint-protocol.md](06-checkpoint-protocol.md) 必填模板的"我的疑虑"段

**好范例**:
> **章节自检**
> - 我的疑虑: §3.2 我假设了客户技术团队有 Python 基础,但访谈纪要里没有验证这一点。如果团队主要是低代码经验,推荐方向 A 的工程量需要重新评估。
> - 需你确认: 客户研发团队的实际技术栈是什么?

## P3 类 · 叙事腔调失真

### P3-1 · 威胁恐吓式机会成本

**触发条件**:写到"为什么必须做"或"不做的代价"时。

**典型表现**:
- "不投资 AI,3 年后损失 5000 万"
- "不上车就落后"
- "做比不做风险更低"

**识别方法**:
```bash
grep -nE "(不投|不做|不上车|机会成本).*?(损失|代价|落后|淘汰)" 主报告.md
```

**修订方案**:
- 删除所有威胁式叙述
- 改用客观叙事: "面临的外部不确定性(政策窗口将关闭 / 竞争对手领先 X 季度 / 现有业务流失风险)"
- 让客户自己得出"不做的代价"的结论,不替客户算账威胁

### P3-2 · 第二人称口语化

**触发条件**:写正文时不自觉用"你"。

**典型表现**:"你需要 ..." / "你会发现 ..." / "你的团队 ..."

**识别方法**:
```bash
grep -nE "(你|您).*[需要|会|的团队|可以]" 主报告.md
```

**修订方案**:
- 全部改为非人称: "许多 X 团队已在使用 ..."、"分析显示 ..."、"客户团队 ..."
- 详见 [05-language-discipline.md §5](05-language-discipline.md)

### P3-3 · 自媒体腔(地震 / 颠覆 / 翻车 / 狂飙)

**触发条件**:写行业变化或竞争描述时。

**典型表现**:
- "行业地震"、"颠覆"、"震动"、"翻车"、"狂飙"
- 标题党风格小标题("一个数字引发的行业地震")

**识别方法**:
```bash
grep -nE "(地震|颠覆|震动|翻车|狂飙|刹车|风暴|血洗)" 主报告.md
```

**修订方案**:
- 改为专业语体: "行业变化"、"结构性调整"、"技术演进"
- 小标题改描述性: "行业背景与 AI 技术成熟度评估"

## P4 类 · 修辞与篇幅

### P4-1 · 高风险词堆砌

**触发条件**:表达加强语气时。

**典型表现**:赋能 / 打造 / 闭环 / 抓手 / 深耕 / 助力 / 沉淀 / 底层逻辑 / 颗粒度 / 心智 / 维度 / 链路 / 生态 / 矩阵 / 范式

**修订方案**:
- 替换为具体动作: "赋能" → "建设/改造/提升"
- 详见 [05-language-discipline.md §2](05-language-discipline.md)

### P4-2 · 排比三连 / 设问开头 / 套话过渡

**触发条件**:段落开头或论证转折时。

**典型表现**:
- "首先 ... 其次 ... 最后 ..."(机械排列)
- "不是 X,不是 Y,不是 Z——而是 W"(排比三连否定)
- "那么,什么是 XXX 呢?"(设问开头)
- "综上所述 / 由此可见 / 不言而喻"(套话过渡)

**识别方法**:
```bash
grep -nE "(首先|其次|最后|综上所述|由此可见|不言而喻|众所周知)" 主报告.md
```

**修订方案**:
- 用逻辑递进替代机械排列
- 删除套话过渡,直接讲结论
- 设问开头改为陈述

### P4-3 · 数量不一致

**触发条件**:写"三个原因"、"五条边界"等数量声明时。

**典型表现**:标题写"三个原因",下面列了四条;标题写"五条边界",下面只列了四条。

**修订方案**:改完内容后必须重新数;标题与列表数量必须严格对应。

## 反模式触发后的处理流程

```
1. 识别(自动 grep + 人工审稿)
2. 标注严重度(P0 / P1 / P2 / P3 / P4)
3. P0 必删 + 改写;P1/P2 必改;P3/P4 视情况修订
4. 修订后回扫所有相关文档(咨询审稿 6 步法 · 第 6 步)
5. 自动 grep 二次验证(期望 0 处命中)
6. 记录到项目质量档案
```

**关键纪律**:反模式不是修一次就完了,是每次主稿改动后都要回查。

## 写完一稿必跑的人工自检 grep 套餐

```bash
# 硬错类

# 检查: 每章散文 ≥ 2 段? 对标格式齐全? 推荐有具象化锚点? 有"不做什么"段? Checkpoint 列了疑虑?
```

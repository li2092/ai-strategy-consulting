# AI 战略报告反模式速查

> 沉淀自多轮外审 + 实战踩坑。每个反模式列触发条件、典型表现、识别方法、修订方案。
> **写之前看**：避免新写出反模式；**写完后看**：用 §scripts 自动检查 + 人工 grep 二次确认。
>
> **关于本文件的自检**：本文件本身大量举"反模式表述"作为反例（例如"千份级文件验证"、"项目组已积累"等）。
> 用 `cross_client_check.py` 扫描本文件**会被预期命中**，这是元文档特性，不是稿件违规。
> 客户实际交付稿件应保持 0 命中。

## 目录

1. P0 类（保密 / 法律 / 事实硬错）—— 4 项
2. P1 类（论证逻辑越级）—— 5 项
3. P2 类（叙事腔调失真）—— 4 项
4. P3 类（修辞与篇幅）—— 3 项

---

## P0 类 · 保密 / 法律 / 事实硬错

### P0-1 · 编造股权链 / 组织关系叙事

**触发条件**：客户为大型企业（国企尤其），有复杂的母子公司关系。

**典型表现**：
- 把"X 集团 → Y 子公司 → Z 客户"的祖孙股权链写成"X 集团是 Z 客户的天然牵头单位"
- 把客户描述为"某央企体系内的牵头单位"而无官方文件支撑
- 编造"客户参与了 X 国标制定"等未经验证的资历

**识别方法**：
```bash
# grep 高风险字眼
grep -nE "母集团|祖集团|牵头单位|天然承担者|参与了.*国标|主导.*标准" 主报告.md
```

**修订方案**：
- 删除所有未经官方股权披露 / 签约证据支撑的"祖孙关系"叙事
- 改为"客户与 X 公司在 Y 业务领域有合作关系"（仅限有公开文件支撑的）
- 资质引用必须查证官方公示，禁止"听说"叙事

**实战案例**：曾把"A 子公司 + B 通用模型"的局部合作外推成"C 集团 → D 母公司 → E 客户"祖孙股权链，事实上 C 与 E 之间并无直接控股关系（控股股东在另一条链路上），被外审判 P0。

### P0-2 · 跨客户内容入稿

**触发条件**：项目组同时服务多个客户，有跨客户的工具 / 数据 / 经验。

**典型表现**：
- 在客户 B 报告中写"客户 A 已验证"、"客户 A 已开发的 X skill"、"千份级文件实战验证"
- 在客户 B 报告中引用客户 A 的内部工具名作为论据
- 把项目组在 A 客户的 KPI 数据当作 B 客户战略的"已验证"基础

**识别方法**：
```bash
# grep 已知客户名 / 工具名
grep -nE "(已验证|实战验证|已开发的|内部已用)" 主报告.md
# 跨客户内容检测脚本
python scripts/cross_client_check.py 主报告.md
```

**修订方案**：
- 删去具名客户与工具，**不要**用"项目组已积累的 X 经验"做替代（这仍是自证）
- 改用三类独立来源：① 公开学术论文 / ② 国际 / 国内标杆公司公开案例 / ③ 政府部委文件
- 工具栈改为开源生态描述（"开源本体工程方案 / 主流 RAG 框架"），不绑具名工具
- 数据规模改为"行业可比规模"或直接删除（不要写"千份级 / 万份级"这类项目组自带的暗号）

**实战案例**：曾在某客户主稿中把"客户 A 已验证"、"客户 A 的某专项 skill"、"千份级文件实战验证"当客户 B 论据，被外审判 P0。同类问题的二次三次重犯几乎都源于"自己用过的工具就是普适证据"的错觉，必须在每段证据写完后做"如果这段交付给第三家客户，还站得住吗"的三问自检。

### P0-3 · 编造数据 / 客户名 / 引文

**触发条件**：写到具体数字 / 客户案例 / 国际公司引文时缺少来源。

**典型表现**：
- "X 公司提升效率 47.3%"（无来源 · 编造的精确百分比）
- "某国际公司官方说：'<虚构的英文原文>'"（无法在该公司官方文档核到的所谓"原文"）
- "某航司用了 AI 降本 8000 万"（具名公司 + 具体数字 + 无来源）

**识别方法**：
```bash
# grep 精确百分比 + 检查是否有来源
grep -nE "[0-9]+\.[0-9]+%" 主报告.md
# grep 英文引文
grep -nE "原文.*['\"].*['\"]" 主报告.md
```

**修订方案**：
- 精确百分比改为区间："47.3%" → "30-50%【推断】"
- 英文引文必须能在官方文档核到，否则改为概括："某国际厂商官方文档将本体定位为 ... 的语义基础层"
- 客户名改为类型：某航司 → "国内某大型枢纽机场"

### P0-4 · 跨客户引用客户案例做对手参照

**触发条件**：列举竞争对手时不小心把"我们也服务过类似客户"暴露。

**典型表现**：
- "我们在为 A 客户服务时发现 ..."（把项目组身份写进了报告）
- "本项目组在 X 行业已积累 N 个案例"（暗示具体数字）

**识别方法**：
```bash
grep -nE "(我们|本项目组|已服务|已积累).*[0-9]+ 个" 主报告.md
```

**修订方案**：
- 改为"国内 X 行业已有可识别的成功落地案例"
- 数量描述改虚："多家"、"若干家"、"N 家"

## P1 类 · 论证逻辑越级

### P1-1 · 自证式外部论据

**触发条件**：项目组成员有公开活动（演讲、博客、文章）。

**典型表现**：
- 把"本项目组顾问在 X 大会的分享"作为"方法论可行性"的外部论据
- 用项目组博客文章佐证"行业共识"

**识别方法**：人工审稿时检查每个【事实】标签的来源是否独立于项目组。

**修订方案**：
- 删除自证式来源
- 改用：① 国际公司官方文档 ② 学术顶会论文 ③ 第三方公开案例 ④ 政府官方文件 ⑤ 行业协会报告

**实战案例**：曾在某客户主稿中把"项目组成员在某行业研学会做主题分享"当方法论可行性依据，被判 P1。项目组自己的公开活动不构成独立外部证据。

### P1-2 · 引文越级（单源 → 行业必做）

**触发条件**：引用某家国际公司的方法论时上升为"行业必做"。

**典型表现**：
- "某国际厂商在过去十年中反复论证：本体是企业 AI 的核心瓶颈"——然后推出"X 客户首期必须建本体层"
- "某 AI 公司 CEO 说 Agent 是未来——所以 X 客户必须现在做 Agent"

**识别方法**：人工审稿检查"X 公司说 Y → 客户必须 Z"这类推理链。

**修订方案**：
- 引用降级为"借鉴"而非"必做"
- 客户首期方案降级为"增强实验"或"条件启动"，附触发条件
- 增加 §3.7.5 类的"方法论应用边界"章节，明确说"借鉴≠复制"

**实战案例**：某稿因单家国际厂商方法论引文 + 增强子模块"必做"结论被判 P1。修订后改为"条件启动"，触发条件是"Week 5-6 纯 RAG 查准率 < N%"——把不可证伪的方法论判断转为可观测的工程门槛。

### P1-3 · 经济模型不当迁移

**触发条件**：引用国际案例时直接搬经济模型。

**典型表现**：
- 用"某国际平台型公司平均合同 N 千万美元"当国内中小预算客户的 ROI 锚点
- 用"美国客户付费水平"推中国国企客户付费能力
- 把通用 SaaS 的 ROI 基准用到工程基建 / 政府类客户

**识别方法**：人工审稿检查 ROI 测算的锚点是否匹配客户实际付费能力。

**修订方案**：
- 客户付费能力按行业重新分档（见 [05-roi-templates.md](05-roi-templates.md)）
- 经济模型迁移时必须扣除"豪华团队 / 全功能平台"的隐含假设
- 加边界声明："国际经验可借鉴，经济模型按本地市场重建"

**实战案例**：某工程基建客户项目原 ROI L2 千万级数字被判定不现实，下调约 80% 后触发"精益启动"重整 —— 决定要因是该客户单项目 IT 投入量级远低于 ROI 模板锚点所对应的客户层级。

### P1-4 · 单一事实源未闭环（活口径残留）

**触发条件**：报告主稿改了，但底稿（如 ROI 测算文档、附件）里旧口径还在。

**典型表现**：
- 同一文件同时说"已作废"又说"正式对外口径以 X 章为准"
- 主报告 §6 写新数字，附件 A1 写旧数字
- 多个文档相互引用，引用链断裂

**识别方法**：
```bash
# grep 矛盾表述
grep -nE "(已作废|历史版本|不作为对外).*?(正式对外|主口径|当前口径)" 全部文档/*.md
# 数字一致性 grep
grep -nE "(投入|回报|ROI).*[0-9]+ 万" 全部文档/*.md
```

**修订方案**：
- 主稿改后回扫所有底稿（咨询审稿 6 步法 · 第 6 步）
- 作废章节加【已作废】标签
- 历史口径明示"原 vX 口径（已作废），现行口径见 [新文件]"

**实战案例**：某项目 ROI 测算文件曾同时存在"本章已作废"和"正式对外口径以 §X 为准"两种表述，被判 P1。修订方案：所有历史口径章节全部加【已作废】标签 + 明示新口径所在文件与章节。

### P1-5 · 战略判断粗化（路径被堵死类表达）

**触发条件**：写竞争分析或技术路线判断时偷懒。

**典型表现**：
- "AI 大模型路径被大厂堵死了"
- "X 行业已经被 Y 覆盖完了"
- "市场没机会了"

**识别方法**：
```bash
grep -nE "(被堵死|被堵|没机会|覆盖完)" 主报告.md
```

**修订方案**：
- 必须分至少三层（基座 / 行业 / 应用）
- 改为："基座层已由 X 占位，行业层 Y 子赛道仍有空间，应用层 Z 类型客户尚未触达"
- 详见 [02-frameworks.md §6 战略判断分层表达](02-frameworks.md#6-战略判断分层表达实战私有--必备)

## P2 类 · 叙事腔调失真

### P2-1 · 威胁恐吓式机会成本

**触发条件**：写到"为什么必须做"或"不做的代价"时。

**典型表现**：
- "不投资 AI，3 年后损失 5000 万"
- "不上车就落后"
- "做比不做风险更低"
- "机会成本 X 千万"

**识别方法**：
```bash
grep -nE "(不投|不做|不上车|机会成本).*?(损失|代价|落后|淘汰)" 主报告.md
```

**修订方案**：
- 删除所有威胁式叙述
- 改用客观叙事："面临的外部不确定性（政策窗口将关闭 / 竞争对手领先 X 季度 / 现有业务流失风险）"
- 让客户自己得出"不做的代价"的结论，不替客户算账威胁

**实战案例**：某项目原 ROI 章节曾用"不投 = 损失 N 千万"威胁，被审核清单判定违规，删除该论证方式 —— 让客户自己得出"不做的代价"才是专业咨询的边界。

### P2-2 · AI 溢价假设

**触发条件**：测算客户外部签约 ROI 时。

**典型表现**：
- "用了 AI 后，每个项目可以多收 10% 溢价"
- "AI 加持后客户愿意多付 20%"

**识别方法**：人工审稿检查 ROI 测算的"额外溢价"假设是否存在。

**修订方案**：
- 删除"AI 溢价"假设（甲方不会因乙方用 AI 让乙方多收钱）
- ROI 算盘子按现行客户单价测，AI 价值体现在内部成本压缩 + 业务量扩大

**实战案例**：某工程基建项目原 ROI 含"现有业务 AI 溢价 10%"子项，被判定不现实并删除 —— 甲方不会因乙方使用 AI 而让乙方多收钱，AI 价值必须体现在内部成本压缩 + 业务量扩大上。

### P2-3 · 第二人称口语化

**触发条件**：写正文时不自觉用"你"。

**典型表现**：
- "你需要 ..."
- "你会发现 ..."
- "你的团队 ..."

**识别方法**：
```bash
grep -nE "(你|您).*[需要|会|的团队]" 主报告.md
```

**修订方案**：
- 全部改为非人称："许多 X 团队已在使用 ..."、"分析显示 ..."、"客户团队 ..."
- 详见 [06-language-discipline.md](06-language-discipline.md)

### P2-4 · 自媒体腔（地震 / 颠覆 / 翻车 / 狂飙）

**触发条件**：写行业变化或竞争描述时。

**典型表现**：
- "行业地震"、"颠覆"、"震动"、"变革"
- "翻车"、"狂飙"、"踩刹车"
- 标题党风格小标题（"一个数字引发的行业地震"）

**识别方法**：
```bash
grep -nE "(地震|颠覆|震动|翻车|狂飙|刹车|风暴|血洗)" 主报告.md
```

**修订方案**：
- 改为专业语体："行业变化"、"结构性调整"、"技术演进"
- 小标题改描述性："行业背景与 AI 技术成熟度评估"

## P3 类 · 修辞与篇幅

### P3-1 · 高风险词堆砌

**触发条件**：表达加强语气时。

**典型表现**：
- 赋能 / 打造 / 闭环 / 抓手 / 深耕 / 助力 / 沉淀 / 底层逻辑 / 颗粒度 / 心智 / 维度 / 链路 / 生态 / 矩阵 / 范式

**识别方法**：
```bash
python scripts/ai_slop_check.py 主报告.md
```

**修订方案**：
- 替换为具体动作："赋能"→"建设/改造/提升"
- 详见 [06-language-discipline.md](06-language-discipline.md)

### P3-2 · 排比三连 / 设问开头 / 套话过渡

**触发条件**：段落开头或论证转折时。

**典型表现**：
- "首先 ... 其次 ... 最后 ..."（机械排列）
- "不是 X，不是 Y，不是 Z——而是 W"（排比三连否定）
- "那么，什么是 XXX 呢？"（设问开头）
- "综上所述 / 由此可见 / 不言而喻"（套话过渡）

**识别方法**：
```bash
grep -nE "(首先|其次|最后|综上所述|由此可见|不言而喻|众所周知)" 主报告.md
```

**修订方案**：
- 用逻辑递进替代机械排列
- 删除套话过渡，直接讲结论
- 设问开头改为陈述

### P3-3 · 数量不一致

**触发条件**：写"三个原因"、"五条边界"等数量声明时。

**典型表现**：
- 标题写"三个原因"，下面列了四条
- 标题写"五条边界"，下面只列了四条
- 数字描述与实际数字不匹配

**识别方法**：人工审稿，写完后数一遍。

**修订方案**：
- 改完内容后必须重新数
- 标题与列表数量必须严格对应

## 反模式触发后的处理流程

```
1. 识别反模式（自动 grep + 人工审稿）
2. 标注严重度（P0 / P1 / P2 / P3）
3. P0 必删 + 改写；P1 必改；P2/P3 视情况修订
4. 修订后回扫所有相关文档（咨询审稿 6 步法 · 第 6 步）
5. 自动 grep 二次验证（期望 0 处命中）
6. 记录到项目质量档案（如审核修订计划文档）
```

**关键纪律**：反模式不是修一次就完了，是每次主稿改动后都要回查。

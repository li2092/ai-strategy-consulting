## 2 · 反 AI 腔

### 2.1 高风险词表(出现即返工)

| 类别 | 禁用词 | 替代策略 |
|---|---|---|
| 万能动词 | 赋能、打造、助力、深耕、驱动、引领、构建、聚焦 | 说具体动作:建设 / 改造 / 提升 / 覆盖 / 开发 / 集中在 |
| 空洞名词 | 闭环、抓手、沉淀、生态、底座、底层逻辑、颗粒度、心智、维度、链路、矩阵、范式 | 说具体事物:反馈机制 / 关键措施 / 知识库 / 合作网络 / 平台 / 推理依据 / 粒度 / 用户认知 / 角度 / 流程 / 对照表 / 模式 |
| 形容词堆砌 | 全面、系统、科学、高效、精准、深入 | 留最准确的一个;删其余 |
| 被动评价 | 被广泛采用、被认为、被誉为、被普遍认为 | 写明谁采用 / 谁评价:例如"BCG 在 2025 年研究中指出" |
| 时代修辞 | 在数字化转型的浪潮中、在 AI 时代的大背景下、随着... 的深入推进 | 直接给具体事实和数据 |

### 2.2 套话过渡禁用

| 禁用 | 替代 |
|---|---|
| 综上所述 | 删除,直接讲结论 |
| 由此可见 | 删除 |
| 不言而喻 | 删除 |
| 众所周知 | 删除(除非接确实"众所周知"的事实,并立即给来源) |
| 值得注意的是 | 删除(如果重要直接写) |
| 不难发现 / 不难看出 | 删除 |
| 总而言之 | 删除 |

### 2.3 排比三连结构禁用

**禁用模式**:
- "首先 ... 其次 ... 最后 ..."(机械排列)
- "不是 X,不是 Y,不是 Z——而是 W"(排比三连否定)
- "X 是 ...,X 是 ...,X 是 ..."(三连排比定义)

**替代**:
- 用逻辑递进替代机械排列:"A 解决 X 问题;A 同时也降低了 Y 成本;进一步看 A 还创造了 Z 资产"
- 用一句话直接说本质:"X 不是营销概念,是经过 N 个客户验证的工程方法"

### 2.4 设问开头禁用

**禁用**:
- "那么,什么是 XXX 呢?"
- "为什么 X 行业需要 AI 呢?"
- "X 的本质到底是什么?"

**替代**:直接陈述定义或判断。

### 2.5 自检 grep

```bash
grep -nE "(赋能|打造|闭环|抓手|深耕|助力|沉淀|底层逻辑|颗粒度|心智|链路|生态|矩阵|范式)" 主报告.md
grep -nE "(综上所述|由此可见|不言而喻|众所周知|值得注意的是)" 主报告.md
grep -nE "(首先.*其次.*最后)" 主报告.md
# 高风险词每处必看,套话过渡和排比三连命中数 = 0
```

## 3 · 反威胁恐吓

### 3.1 禁用模式

| 模式 | 例子 | 替代 |
|---|---|---|
| 机会成本威胁 | "不投资,3 年损失 5000 万" | "面临的外部不确定性:政策窗口将关闭 / 竞争对手领先 X 季度 / 业务流失风险" |
| 时机焦虑 | "不上车就落后 / 现在不做就没机会了 / 错过这个窗口就永远追不上" | "当前政策窗口期为 X 季度;超过该窗口需重新评估进入策略" |
| 比较恐吓 | "做比不做风险更低 / 不做才是最大的风险" | 删除该论证方式(让客户自行判断) |
| 末日叙事 | "如果 X,公司将面临生存危机" | "X 情况下公司在 Y 业务上的市场地位将面临压力" |

### 3.2 可以保留的客观表述

威胁与客观影响的区别:
- **威胁**:替客户算账 + 强调代价 + 暗示焦虑
- **客观**:陈述事实 + 让客户自行判断 + 不渲染情绪

**客观表述模板**:
> "若 X 情况发生,对客户业务产生以下影响:
> - 影响 1: [具体业务/数字/时间]
> - 影响 2: [具体业务/数字/时间]
> - 影响 3: [具体业务/数字/时间]
> 该影响的处理方案见 §X.Y。"

### 3.3 自检 grep

```bash
grep -nE "(不投|不做|不上车|机会成本|不上车就|做比不做|时不我待|生存危机)" 主报告.md
# 期望命中数 0 处
```

## 4 · 反自说自话

### 4.1 自证式论据禁用

**禁用模式**:把项目组自己的活动 / 文章 / 工具当作"外部已验证"的论据。

**典型例子**:
- "本项目组顾问在 X 大会的分享 → 证明方法论可行"
- "项目组博客已论证 X → 行业共识"
- "项目组开发的 Y skill → 已实战验证"

**替代来源**(按可信度排序):
1. **政府官方文件** —— 部委发文、白皮书
2. **国际公司官方文档** —— Anthropic / OpenAI / Google / Microsoft 等公开技术博客与文档
3. **学术顶会论文** —— ACL / NeurIPS / VLDB / ICML 等
4. **第三方研究报告** —— BCG / McKinsey / MIT Sloan / IDC / Gartner
5. **公开客户案例** —— Customer Story 公开页
6. **行业协会数据** —— 行业协会发布的白皮书

### 4.2 跨客户内容入稿禁用

**禁用模式**:把 A 客户的内部工具 / 数据 / 已验证成果写进 B 客户报告。

**典型例子**:
- "X 公司已开发的 Y skill"(具名客户 + 具名工具)
- "已经过 N 个客户千份级文件验证"(隐含跨客户经验)
- "本项目组在 X 行业已积累 N 个案例"(暴露跨客户身份)

**修订方法**:
- 删除具名客户 + 工具
- 改为开源工具栈描述,不绑具名工具
- 数量描述改虚:"多家"、"若干家"

### 4.3 写之前的三问自检

每段写完检查:
1. 这段证据是否属于本客户?
2. 是否是独立于项目组的外部来源?
3. 如果交付给第三家客户,这段话还站得住吗?

三问中任一"否" → 删除或改写。

### 4.4 自检 grep

```bash
grep -nE "(已验证|实战验证|已开发的|内部已用)" 主报告.md
grep -nE "(我们|本项目组|已服务|已积累).*[0-9]+ ?个" 主报告.md
# 期望命中数 0 处
```

## 5 · 反口语化

### 5.1 第二人称禁用

| 禁用 | 替代 |
|---|---|
| 你 / 你的 / 你们 | 客户 / 客户团队 / 企业团队 / 相关方 |
| 你需要... | 需要 / 建议 / 应当 |
| 你会发现... | 分析显示 / 数据表明 |
| 你的团队 | 客户团队 / 业务团队 |

### 5.2 自媒体腔禁用

| 类别 | 禁用 | 替代 |
|---|---|---|
| 灾难修辞 | 地震、颠覆、震动、变革、风暴、血洗 | 行业变化 / 结构性调整 / 技术演进 |
| 速度修辞 | 翻车、狂飙、踩刹车、爆发、井喷 | 失败 / 快速推进 / 调整策略 / 增长 / 增速加快 |
| 情感修辞 | 信号很明确 / 答案很清楚 / 一目了然 | 由此可以判断 / 基于以上分析 |
| 标题党 | "一个数字引发的行业地震" | 描述性标题:"行业背景与 AI 技术成熟度评估" |
| 悬念 | "故事还没有结束" / "但这只是开始" | 直接陈述完整发现 |

### 5.3 假亲切禁用

**禁用**:
- "本能地" / "自然而然" / "你会发现" / "我们都知道" / "懂的人都懂"

**替代**:直接陈述事实或判断。

### 5.4 自检 grep

```bash
grep -nE "(你|您).*(需要|会|的团队|可以)" 主报告.md
grep -nE "(地震|颠覆|震动|变革|翻车|狂飙|井喷|血洗)" 主报告.md
grep -nE "(本能地|自然而然|懂的人都懂)" 主报告.md
# 期望全部 0 处
```

## 6 · 三标签纪律

### 6.1 标签定义

| 标签 | 含义 | 示例 |
|---|---|---|
| 【事实】 | 有公开数据 / 文件 / 案例支撑 | "Anthropic 2024-10 发布 Claude 3.5 Sonnet New【事实】" |
| 【推断】 | 基于事实的合理外推,写明推理链 | "客户某业务工程师约 60 人【推断】(基于年报披露的总员工 X + 业务收入占比)" |
| 【假设】 | 需客户内部验证,写明触发条件 | "AI 加持后工时节约 30%【假设 · MVP 完成后回归验证】" |

### 6.2 必标场景

下列每个判断都必须带标签,少一个就要返工:
- 任何具体数字(人数 / 收入 / 占比 / ROI / 工时节约 / 客户单价)
- 任何因果关系("X 导致 Y")
- 任何对手 / 客户行为预测
- 任何时间窗判断("3 年内 X 将发生")
- 任何技术成熟度判断

### 6.3 标签的层级

**禁止**:
- 把【推断】写成【事实】(过度自信)
- 把可证伪的结论标【假设】(推卸责任)
- 一个数字混合多种来源后只标一个标签

**正确**:
- 多源融合的数字标主标签 + 注脚解释来源构成
- 不确定时标【推断】(默认偏保守)

### 6.4 自检方法

人工审稿:每个数字 / 因果 / 预测都要找到对应的【事实】/【推断】/【假设】标签。
重点章节(执行摘要、Top 推荐、路线图)的标签覆盖率应 ≥ 85%。

## 7 · 句式与段落控制

### 7.1 句子长度

| 类型 | 长度 | 处理 |
|---|---|---|
| 短句 | < 15 字 | 主体;可连用 |
| 中句 | 15-25 字 | 主体;可连用 |
| 长句 | 25-40 字 | 视情况,需有清晰断点 |
| 超长句 | > 40 字 | 必须拆 |

### 7.2 段落长度

| 类型 | 长度 | 处理 |
|---|---|---|
| 短段 | < 100 字 | 适合开头 / 收尾 / 强调 |
| 中段 | 100-200 字 | 主体(最常用) |
| 长段 | 200-300 字 | 论证密集时 |
| 超长段 | > 300 字 | 必须拆 |

### 7.3 信息密度

每句都要有新信息。**自检法**:删掉这句话整段是否仍能理解?能 = 这句该删。

### 7.4 同义词控制

同一概念全文统一术语,**不允许同义词漂移**。

**禁用**:同段中"Agent / 智能体 / AI Agent / AI 助手"混用。

**正确**:第一处定义"Agent(智能体)",全文统一用"Agent"。

### 7.5 段落微结构 · Claim-Evidence-Implication

每个分析型段落都按"主张 → 证据 → 含义"三句套路展开,缺一即返工。

| 元素 | 写什么 | 在段中的位置 |
|---|---|---|
| **Claim 主张** | 你判断什么(首句直接给结论) | 段首 1 句 |
| **Evidence 证据** | 数据 / 案例 / 来源 / 推理 | 中段 1-3 句 |
| **Implication 含义** | "所以呢?" 这意味着客户该怎么决策 | 段末 1 句 |

**示例**(拆解一个合规段落):

> 客户当前 X 业务的工程师单日规范查询时间约 30-60 分钟【推断】**[Claim:现状有提效空间]**。基于行业 N 家同类企业的工时调研,该值占工程师全日工时的 7-12%,且呈现"高频低值"特征 —— 单次查询任务难度低但中断深度工作流程**[Evidence:数据 + 模式]**。这构成 AI Copilot 的典型切入场景:用秒级响应替代分钟级查询,可在 6-10 周内验证 30%+ 提效**[Implication:so what]**。

**反例**(缺失 Implication):

> 客户工程师单日规范查询约 30-60 分钟。这个值很高。 ❌ 没说"所以"

**反例**(缺失 Evidence):

> 客户应当上 AI Copilot,因为效率会大幅提升。 ❌ 主张和含义都有,证据呢?

## 8 · 标点与术语规范

### 8.1 中文标点

- 中文段用全角:,。;:?!""''()
- 英文段用半角:, . ; : ? ! "" '' ()
- **不允许**:中英文标点混用(如中文段用半角逗号)

### 8.2 英文术语

- 中英混排时英文用空格隔开:"采用 LoRA 技术",不是"采用LoRA技术"
- 数字 + 单位之间不空:"5 万"或"5万"(统一一种风格,不混用)
- 缩写首次定义:"Forward Deployed Engineer(FDE,前沿部署工程师)"

### 8.3 章节编号

公文标准(中文报告):
- 一级:一、二、三、四 ...(或 第 1 章 / 第 2 章 ...)
- 二级:(一)(二)(三)(或 1.1 / 1.2 / 1.3 ...)
- 三级:1. 2. 3.(或 1.1.1 / 1.1.2 ...)

### 8.4 列表与表格

- 项目符号:- 或 •(不用 ✓ ★ 等花哨符号)
- 表格表头加粗
- 长表格分页或拆为多个小表

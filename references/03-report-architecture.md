## 2 · 第 0 章 · 名词与术语速览

**目标**：为非技术读者提供基础术语索引，避免在 §1 起就被术语打断。

**结构**：分类列表，每条 1 行 `术语（英文，缩写）：一句话定义`。

**典型分类**：
- AI 基础（LLM / RAG / Agent / MCP / Copilot 等）
- 模型与微调（Qwen / DeepSeek / GLM / LoRA / 蒸馏）
- 行业专有（视客户行业 · 如 BIM / IFC / 桥隧专项 等）
- 政策与事件（92 号文 / 1+N+X / 各类大赛）
- 时间编号（M1-M12 / Year 1-3）

**纪律**：
- 总长 ≤ 1 页
- 不做叙事，纯名词索引
- 避免与 §1 重复内容
- 不写"什么是 AI / 什么是大模型"这类基础百科（读者不是小白）

## 3 · 第 1 章 · 执行摘要

**目标**：让董事长读完 §1 就能拍板"是否继续读后面"。

**结构**：SCQA + Claim-Evidence-Implication

```
1.1 战略判断（核心建议 · 1 段，3-5 句）
1.2 战略空间（为什么是这个空间 · 三圆相交结构）
1.3 战略主线（双产品线 / 单产品线骨架 · 1 段简述）
1.4 投入产出（三档投入 + 综合 ROI · 一段数字）
1.5 启动建议（第一步 · 要批准什么）
```

**关键纪律**：
- 第一句话不是"在这个数字化转型的时代"，而是"综合对 X 行业 AI 演进节奏 / 政策窗口 / 客户禀赋的评估，本报告建议..."
- 禁用威胁恐吓（"不投资 = 损失 X 万"）
- 三标签必须用（每个数字带【事实/推断/假设】）
- 最后一段必须是"启动建议 + 决策点"，不是"展望与愿景"

**反例**（必须避开）：
> "随着人工智能技术的飞速发展，企业数字化转型已经成为不可逆转的潮流。X 公司作为行业领军者，必须把握 AI 转型的历史机遇..."

**正例**（通用结构 · 具体数字按客户实际填写）：
> "综合对 X 行业 AI 演进节奏、相关部委政策窗口与客户自身禀赋的评估，本报告建议客户在政策窗口启动后 N 个月内启动 Y 双产品线战略。三项约束在该时间窗共同收敛，均为外部客观条件，与企业意愿无关。基准档三年投入约 X 万元，综合 ROI 约 N%，回本期 N1 个月（综合）/ N2 个月（现金）。第一步是 6-10 周 MVP 验证，外部支出 N3 万元，内部抽调 M 名骨干工程师..."

### 3.1 开篇钩子 4 模式（相邻报告不允许重复同一钩子）

| 模式 | 用在何时 | 模板示例 |
|---|---|---|
| **数据钩子** | 手里有冲击性的数字 | "X 行业每年产生 N 亿条 Y 数据，其中 M% 未被结构化利用 —— 折算 ¥X 亿沉没成本。" |
| **张力钩子** | 存在悖论或矛盾 | "客户在 X 投入 ¥M 亿做数字化，但客户满意度反而下降 N pp。" |
| **判决钩子** | 结论反直觉 | "问题不在技术。客户已有 N 套 AI 系统，问题是它们之间没有数据流通。" |
| **场景钩子** | 用一个具体场景把问题画给读者看 | "周一早上 9 点，工程师打开 X 系统查 Y 规范 —— 这一动作每周重复 N 次，全公司每年消耗 M 万小时。" |

**4 类禁用开篇**（出现即返工）：

- ❌ "随着 X 的快速发展 / 在数字化转型的浪潮中 ..."
- ❌ "什么是 AI 战略？为什么我们需要它？" （设问开头）
- ❌ "正如 Peter Drucker 所说 ..." （借名人开场）
- ❌ "本报告将分为以下章节 ..." （目录式开场）

## 4 · 第 2 章 · 战略环境

**目标**：建立"为什么是现在 + 为什么是这条路"的客观依据。

**结构**：
```
2.1 政策环境（部委文件 + 时间窗 + 强制/激励/窗口三类影响）
2.2 行业现状（规模 + 增速 + 价值链 + 数字化成熟度）
2.3 竞争格局（必须分基座/行业/应用三层）
2.4 客户禀赋（资质 + 资产 + 数据 + 决策链）
2.5 战略空间（前四项的交集）
```

**关键画面**：
- **政策时间窗甘特图**：纵轴政策项，横轴时间，标注窗口期与考核节点
- **竞争格局分层图**：三层（基座/行业/应用），每层标注头部占位与剩余空间

**关键纪律**：
- §2.3 竞争格局必须分层（铁律 5 · 分层表达），禁止"路径被堵死"
- §2.4 客户禀赋只列已公开/已验证的，不编造（铁律 6 · 认知诚实）

## 5 · 第 3 章 · 战略主线

**目标**：把 §2 的战略空间转化为可执行的产品 + 能力 + 业务 + 生态四层结构。

**结构**：
```
3.1 战略主线总览（四层架构图）
3.2 产品层（双产品线 / 单产品线 + 协同关系）
3.3 能力层（共享底座 L1-L6 + 特化子层）
3.4 业务层（V 形聚焦 + 锚点专业）
3.5 生态层（合作伙伴 + 接口 + 可插拔策略）
3.6 关键技术决策（开源 vs 自研 / 云 vs 本地 / 国产 vs 国际）
3.7 设计方法论（视情况选填 · 如本体驱动设计、Agent 编排范式等）
3.8 本章小结
```

**关键画面**：
- **四层架构总图**（最重要 · §3.1）：横向四层，每层标注核心组件 + 共享/特化区分

**关键纪律**：
- §3.2 产品命名必须有"一句话定位"（"X 是 Y 形态的 Z 应用"）
- §3.3 能力层避免技术堆砌，每层都要回到"解决什么问题"
- §3.6 关键技术决策必须列"做 vs 不做"对照表
- §3.7 方法论章节如果引用国际公司（如 Palantir / OpenAI），必须用官方公开文档可核表述，禁止编造引文

## 6 · 第 4 章 · 首发场景设计

**目标**：把战略主线落到 1-2 个具体场景上，让客户看到"长什么样"。

**结构**（每个产品线一节）：
```
4.X.1 场景描述（用户故事 · 现状 vs 改造后）
4.X.2 关键差异化（vs 竞品 · 三点）
4.X.3 价值四问（解决什么 + 提效多少 + 扩什么市场 + 加速什么客户体验）
4.X.4 三层价值承接（L1 + L2 + L3 · L4 视情况）
4.X.5 技术实现简述（不深入，详见 §5 MVP 路径）
```

**关键画面**：
- **工作流前后对比图**：左侧"现状"用户旅程，右侧"AI 后"用户旅程

**关键纪律**：
- 每个场景必须过价值四问，过不了的不写进 §4
- 价值承接 L1+L2+L3+L4 多于一层加分

## 7 · 第 5 章 · MVP 路径

**目标**：给出 6-10 周的可执行试点计划。

**结构**：
```
5.1 MVP 设计原则（双门槛 / 双产品线并行 / 不超预算 / 可观测可调头）
5.2 Week 1-10 时间表（含每周任务、产出、参与角色）
5.3 MVP 通过标准（量化门槛）
5.4 MVP 后续（通过 → Phase 2 / 不通过 → 回归原因 + 决策选项）
5.5 风险与缓释
```

**关键画面**：
- **Week 1-10 时间表**：横轴周次，纵轴产品线，单元格写每周任务

**关键纪律**：
- MVP 周期必须 ≤ 10 周（精益启动），超过则改为 Phase 1A + Phase 1B
- 必须有"不通过 → 怎么办"的退出条件，不能默认必通过
- 双门槛通常是"查准率 80% / 检出率 80%" + "工时节约 30%" 这类硬数字

## 8 · 第 6 章 · 投入产出（关键章节）

**目标**：给出精益启动方案的三档投入 × 四层回报矩阵 + 综合 ROI。

**结构**：
```
6.1 投入测算（三档 · 保守 / 基准 / 激进 · 含投入分类）
6.2 客户与定价（三档客户单价 + 三年签约漏斗）
6.3 回报测算（L1 内部提效 + L2 外部签约 + L3 硬资产 + L4 软实力）
6.4 综合 ROI（现金 ROI + 综合 ROI + 回本期）
6.5 敏感性分析（关键变量上下浮动 ±20%-50% 的 ROI 区间）
6.6 一句话口径（对董事长）
```

**关键画面**：
- **三档投入 × 四层回报矩阵**（最重要 · §6.4）：左侧三档投入，右侧四层回报，中间箭头连接

**关键纪律**：
- 三档不能并列展示（让客户挑等于无主见）—— 必须选定基准档为主口径，保守和激进作"风险/上行"两条侧线
- 所有数字必须三标签（【事实/推断/假设】）
- 禁止"机会成本威胁"叙事（"不投 = 损失 X 万"）
- L4 软实力必须独立列出，不混入 ROI 计算
- 详细模板见 [05-roi-templates.md](05-roi-templates.md)

## 9 · 第 7 章 · 三年里程碑

**目标**：把战略落到月度时间轴上，明确每个节点的输出与决策点。

**结构**：
```
7.1 Year 1 月度里程碑（M1-M12 · 双产品线 + 大赛申报 + 内部三业务落地）
7.2 Year 2 季度里程碑（外部签约 + 产品扩展 + 政策红利获取）
7.3 Year 3 半年度里程碑（规模化 + 跨行业扩展 + 平台化）
7.4 关键决策门（董事长签字节点）
```

**关键画面**：
- **三年甘特图**：横轴时间，纵轴工作流（产品 / 销售 / 技术 / 大赛 / 政策对接）

**关键纪律**：
- M1 必须是 MVP 启动月，M3 是 MVP 收官月
- 每个里程碑标注"通过条件"（不通过怎么办）
- 决策门标注"董事长签字"vs"项目经理裁定"两类

## 10 · 第 8 章 · 风险与应用边界

**目标**：列清楚"这个战略可能怎么死 + 我们决定不做的事"。

**结构**：
```
8.1 风险三类（技术 / 组织 / 市场 · 各 3-5 项）
8.2 应用边界（这个方法论的适用范围 · N 条边界）
8.3 不做什么 · N 件（每件带理由 + 替代方案）
8.4 退出条件（什么情况下停止 / 调整）
```

**关键画面**：
- **风险三类雷达图**：技术 / 组织 / 市场三轴，每轴 3-5 个具体风险
- **红线表**：8 项不做的事 + 理由 + 替代

**关键纪律**：
- "不做什么"必须 ≥ 8 项，少于 8 项 = 没做判断力
- 每条不做必须带理由 + 替代方案
- 退出条件必须可观测（量化指标），不能"领导主观判断"

## 11 · 第 9 章 · 组织保障与人员

**目标**：解决"谁来做 + 怎么管"的问题。

**结构**：
```
9.1 组织原则（Agent 优先 / 人承担三类核心 · 技术底座 / 产品判断 / 业务测试）
9.2 关键岗位（首席架构师 / 产品经理 / 业务测试负责人 / 战略枢纽 等）
9.3 人员规模与曲线（Year 1 / 2 / 3 各阶段）
9.4 与现有岗位的关系（替代 / 协同 / 新增）
9.5 招聘与培养策略
9.6 激励与考核机制
```

**关键画面**：
- **三年人员曲线**：横轴时间，纵轴人数，对比"传统模式"vs"AI 加持模式"
- **核心岗位分工矩阵**：行=岗位，列=职责，单元格=负责度

**关键纪律**：
- 必须强调"Agent 优先"原则（小团队 + AI 加持 vs 传统大团队）
- 不必为每个岗位写四维能力详述（实施层细节，留 Q&A）
- 薪酬级别不写具体数字（"对标 X 级别"会被判定过实），只说"按客户既有制度"

## 12 · 第 10 章 · 启动建议与决策点

**目标**：明确告诉董事长今天要批准的事。

**结构**：
```
10.1 启动建议（一句话）
10.2 决策清单（3-5 项 · 含金额 + 时间窗 + 授权对象 + 风险敞口）
10.3 不批准的代价（客观叙述，不威胁恐吓）
10.4 一页纸（如有 · 链接到附件 / 单独文档）
```

**关键画面**：
- **决策清单卡**：3-5 项决策，每项 4 列（金额 / 时间窗 / 授权对象 / 风险敞口）

**关键纪律**：
- 决策必须具体到"批准 X 万 MVP 试点 / 批准战略枢纽招聘授权 / 批准 Year 1 X 万框架"
- 不允许"请董事长指示"这种虚收尾
- 不批准的代价用客观语言（"政策窗口将关闭 / 竞争对手领先 X 季度"），不用威胁语言（"不批就死定了"）
